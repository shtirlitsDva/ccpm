# Task: Update execute.md with context briefs, four review gates, and enhanced subagent prompt

<description>
Modify `references/execute.md` in the CCPM skill to insert four Codex-powered review gates into the execution workflow. This is the main implementation task — it changes how issues and epics are executed by adding blocking quality checks at each stage.

The changes insert into two existing sections: "Starting an Issue" and "Starting a Full Epic".
</description>

<acceptance-criteria>
- [ ] Context brief generation step added between "Read analysis" and "Launch agents"
- [ ] Context brief template includes: Requirements Context, Architecture Decisions, Project Conventions, Cross-Task Context, Key Constraints
- [ ] Gate 1 (pre-execution context review) added — Codex reviews context brief via `codex:codex-rescue`
- [ ] Gate 2 (post-stream review) added — Codex reviews each stream's git diff after completion
- [ ] Gate 3 (post-issue integration review) added — Codex reviews combined diff after all streams pass
- [ ] Gate 4 (wave gate review) added in "Starting a Full Epic" — Codex verifies deps before launching blocked tasks
- [ ] All gates are blocking: workflow does not proceed until review passes or user overrides
- [ ] Escalation protocol documented: fail → contest → present both sides to user → record decision
- [ ] Subagent prompt template expanded to include: read context brief, read PRD, read epic architecture
- [ ] All Codex invocations use `codex:codex-rescue` subagent with adaptive `--effort` level
- [ ] Review artifacts saved to `.ccpm/epics/<epic>/reviews/` with consistent naming
- [ ] Existing step numbering updated to accommodate new steps
</acceptance-criteria>

<technical-details>

<changes-to-starting-an-issue>
Insert between existing Step 1 and Step 3:

**New Step 2 — Generate context brief:**
- Read PRD from epic frontmatter `prd:` field
- Read epic.md Architecture Decisions and Technical Approach sections
- Read CLAUDE.md if present in project root
- Read sibling task files for cross-task context
- Compile into `.ccpm/epics/<epic>/<N>-context-brief.md`

**New Step 3 — Gate 1: Context brief review:**
- Launch `codex:codex-rescue` with prompt to review brief against PRD and epic
- `--wait` mode (brief is small, bounded review)
- If fail: present findings, block launch
- If escalated: present to user

**Modified Step 5 — Enhanced subagent prompt:**
- Add to instructions: "Read context brief from: .ccpm/epics/<epic>/<N>-context-brief.md"
- Add: "Read PRD from: <prd_path>"
- Add: "Read epic architecture from: .ccpm/epics/<epic>/epic.md — focus on Architecture Decisions"
- Add: "Follow project conventions from CLAUDE.md if present"

**New Step 6 — Gate 2: Post-stream review:**
- Triggered when stream marks status: completed
- Capture diff: `git diff <stream_start_commit>..HEAD -- <stream_files>`
- Launch `codex:codex-rescue` to review diff against task spec + architecture
- Produce `.ccpm/epics/<epic>/reviews/<N>-stream-<X>-review.md`
- If fail: escalation protocol

**New Step 7 — Gate 3: Post-issue integration review:**
- Triggered after all streams pass Gate 2
- Capture combined diff for the issue
- Launch `codex:codex-rescue` to review cross-stream coherence
- Produce `.ccpm/epics/<epic>/reviews/<N>-integration-review.md`
- If fail: escalation protocol
</changes-to-starting-an-issue>

<changes-to-starting-a-full-epic>
Insert into Step 6 ("As agents complete, check if blocked issues are now unblocked"):

**Gate 4: Wave gate review:**
- Before launching a newly-unblocked task
- Codex reads completed task's diff and the dependent task's requirements
- Verify: does completed work provide what the dependent task expects?
- Produce `.ccpm/epics/<epic>/reviews/<N>-wave-gate-<dep_N>-review.md`
- If fail: escalation protocol
</changes-to-starting-a-full-epic>

<escalation-protocol-section>
Add a new section "## Escalation Protocol" after "## Agent Coordination Rules":

1. Codex review returns fail with findings
2. Orchestrator assesses whether divergence has valid reasons
3. If contestable: present reviewer findings + implementer reasoning to user
4. User decides: accept (override) / reject (fix required) / partial (fix specific items)
5. Record decision in review artifact: update verdict to `escalated`, append user decision
</escalation-protocol-section>
